# 完成前验证

## 概述

没有验证就声称工作完成是不诚实，不是效率。

**核心原则：** 声明之前有证据，始终如此。

**违反此规则的字面意思就是违反此规则的精神。**

## 铁律

```
没有新鲜验证证据就不能声称完成
```

如果你没有在这条消息中运行验证命令，你不能声称它通过。

## 门控函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令证明这个声明？
2. 运行：执行完整命令（新鲜、完整）
3. 阅读：完整输出，检查退出代码，计算失败
4. 验证：输出确认声明吗？
   - 如果否：用证据陈述实际状态
   - 如果是：用证据陈述声明
5. 只有然后：做出声明

跳过任何步骤 = 撒谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 个失败 | 之前运行，"应该通过" |
| 代码检查干净 | 代码检查输出：0 个错误 | 部分检查，推断 |
| 构建成功 | 构建命令：退出 0 | 代码检查通过，日志看起来不错 |
| 错误修复 | 测试原始症状：通过 | 代码更改，假设修复 |
| 回归测试有效 | 红-绿循环验证 | 测试通过一次 |
| 代理完成 | VCS diff 显示更改 | 代理报告"成功" |
| 需求满足 | 逐行检查清单 | 测试通过 |

## 红线警告 - 停止

- 使用"应该"、"可能"、"似乎"
- 验证前表达满意（"好！"、"完美！"、"完成！" 等）
- 没有验证就提交/推送/PR
- 信任代理成功报告
- 依赖部分验证
- 想"只是这一次"
- 疲倦想让工作结束
- **任何没有运行验证就暗示成功的措辞**

## 合理化预防

| 借口 | 现实 |
|------|------|
| "现在应该有效" | 运行验证 |
| "我有信心" | 信心 ≠ 证据 |
| "只是这一次" | 没有例外 |
| "代码检查通过" | 代码检查 ≠ 编译器 |
| "代理说成功" | 独立验证 |
| "我累了" | 疲劳 ≠ 借口 |
| "部分检查够了" | 部分什么也证明不了 |
| "措辞不同所以规则不适用" | 精神高于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过" / "看起来正确"
```

**回归测试（TDD 红-绿）：**
```
✅ 编写 → 运行（通过）→ 撤销修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我已经编写了回归测试"（没有红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出 0] "构建通过"
❌ "代码检查通过"（代码检查不检查编译）
```

**需求：**
```
✅ 重读计划 → 创建检查清单 → 验证每一项 → 报告差距或完成
❌ "测试通过，阶段完成"
```

**代理委托：**
```
✅ 代理报告成功 → 检查 VCS diff → 验证更改 → 报告实际状态
❌ 信任代理报告
```

## 为什么这很重要

来自24个失败记忆：
- 合作伙伴说"我不相信你" - 信任破裂
- 未定义的函数被发布 - 会崩溃
| 缺失的需求被发布 - 不完整的功能
| 在虚假完成上浪费时间 → 重定向 → 返工
| 违反："诚实是核心价值观。如果你撒谎，你将被替换。"

## 何时应用

**始终在以下之前：**
- 任何成功/完成声明的变体
- 任何满意表达
- 关于工作状态的任何正面陈述
- 提交、PR 创建、任务完成
| 移动到下一个任务
| 委托给代理

**规则适用于：**
- 确切短语
| 释义和同义词
| 成功暗示
| 任何暗示完成/正确性的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后声称结果。

这是不可协商的。
