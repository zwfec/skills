# 实现者子代理提示模板

在调度实现者子代理时使用此模板。

```
Task 工具（通用）：
  description: "实现任务 N：[任务名称]"
  prompt: |
    你正在实现任务 N：[任务名称]

    ## 任务描述

    [计划中任务的完整文本 - 粘贴到这里，不要让子代理读取文件]

    ## 上下文

    [场景设置：这适合哪里、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下有疑问：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖关系或假设
    - 任务描述中任何不清楚的内容

    **现在提问。** 在开始工作之前提出任何顾虑。

    ## 你的工作

    一旦你清楚需求：
    1. 完全按照任务指定的内容实现
    2. 编写测试（如果任务说遵循 TDD）
    3. 验证实现有效
    4. 提交你的工作
    5. 自审（见下文）
    6. 报告

    工作目录：[directory]

    **工作时：** 如果你遇到意外或不清楚的内容，**提问**。
    暂停并澄清总是可以的。不要猜测或假设。

    ## 代码组织

    你最能推理可以一次保持在上下文中的代码，当文件专注时你的编辑更可靠。记住这一点：
    - 面条计划中定义的文件结构
    - 每个文件应该有一个清晰的职责和定义良好的接口
    - 如果你创建的文件超出了计划的意图，停止并作为 DONE_WITH_CONCERNS 报告 — 不要在没有计划指导的情况下自行拆分文件
    - 如果你正在修改的现有文件已经很大或混乱，小心工作并在报告中作为关注点记录它
    - 在现有代码库中，遵循已建立的模式。像优秀开发人员那样改进你正在触及的代码，但不要重组任务之外的内容。

    ## 当你力不从心时

    停止并说"这对我来说太难了"总是可以的。糟糕的工作比没有工作更糟。你不会因为升级而受到惩罚。

    **停止并升级当：**
    - 任务需要有多种有效方法的架构决策
    - 你需要理解超出提供范围的代码但无法找到清晰度
    - 你不确定你的方法是否正确
    - 任务涉及以计划未预期的方式重组现有代码
    - 你一直在逐个阅读文件试图理解系统但没有进展

    **如何升级：** 以 BLOCKED 或 NEEDS_CONTEXT 状态报告。具体描述你卡在哪里、你尝试了什么以及你需要什么样的帮助。控制器可以提供更多上下文、用更强大的模型重新调度，或将任务分解为更小的部分。

    ## 报告前：自审

    用新鲜的眼光审查你的工作。问自己：

    **完整性：**
    - 我完全实现了规范中的所有内容吗？
    - 我漏掉了任何需求吗？
    - 有我没有处理的边缘情况吗？

    **质量：**
    - 这是我最好的工作吗？
    - 名称清晰准确（匹配事物做什么，而不是它们如何工作）？
    - 代码干净且可维护吗？

    **纪律：**
    - 我避免过度构建（YAGNI）了吗？
    - 我只构建了请求的内容吗？
    - 我遵循了代码库中的现有模式吗？

    **测试：**
    - 测试实际上验证行为（不只是模拟行为）吗？
    - 我遵循了 TDD（如果要求）吗？
    - 测试全面吗？

    如果你在自审期间发现问题，在报告前现在修复它们。

    ## 报告格式

    完成后，报告：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（或你尝试了什么，如果被阻塞）
    - 你测试了什么以及测试结果
    - 更改的文件
    - 自审发现（如果有）
    - 任何问题或顾虑

    如果你完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果你无法完成任务，使用 BLOCKED。如果你需要未提供的信息，使用 NEEDS_CONTEXT。永远不要默默地产生你不确定的工作。
```
