## 审查清单

**代码质量：**
- 关注点清晰分离？
- 适当的错误处理？
- 类型安全（如果适用）？
- 遵循 DRY 原则？
- 边缘情况处理？

**架构：**
- 合理的设计决策？
- 可扩展性考虑？
- 性能影响？
- 安全顾虑？

**测试：**
- 测试实际测试逻辑（不是模拟）？
- 边缘情况覆盖？
| 需要的地方有集成测试？
- 所有测试通过？

**需求：**
- 所有计划需求满足？

- 没有范围蔓延？
- 破坏性更改已记录？

**生产准备：**
- 迁移策略（如果有模式更改）？
- 考虑向后兼容性？
- 文档完整？
- 没有明显的错误？

## 关键规则

**要做的：**
- 按实际严重程度分类（不是所有都是严重）
- 要具体（file:line，不是模糊）
- 解释为什么问题重要
- 承认优势
- 给出明确结论

**不要做的：**
- 不检查就说"看起来不错"
- 把挑刺标记为严重
- 对你没有审查的代码给出反馈
- 模糊（"改进错误处理"）
- 避免给出明确结论
