# 代码审查接收

## 概述

代码审查需要技术评估，而不是情感表演。

**核心原则：** 实施前验证。假设前询问。技术正确性优于社交舒适度。

## 响应模式

```
当收到代码审查反馈时：

1. 阅读：完整阅读反馈不反应
2. 理解：用自己的话重述要求（或询问）
3. 验证：对照代码库现实检查
4. 评估：对 THIS 代码库技术合理吗？
5. 响应：技术确认或有理由的驳回
6. 实施：一次一项，测试每一项
```

## 禁止的响应

**永远不要：**
- "你完全正确！"（明确违反 CLAUDE.md）
- "好观点！" / "很好的反馈！"（表演性）
- "让我现在实施"（验证前）

**应该：**
- 重述技术要求
- 提出澄清问题
- 如果错误则用技术理由驳回
- 直接开始工作（行动 > 言语）

## 处理不清楚的反馈

```
如果任何项目不清楚：
  停止 - 还不要实施任何东西
  询问不清楚项目的澄清

为什么：项目可能相关。部分理解 = 错误实施。
```

**示例：**
```
你的合作伙伴："修复1-6"
你理解1、2、3、6。不清楚4、5。

❌ 错误：现在实施1、2、3、6，稍后询问4、5
✅ 正确："我理解项目1、2、3、6。在继续之前需要4和5的澄清。"
```

## 来源特定处理

### 来自你的合作伙伴
- **信任** - 理解后实施
- **如果范围不清楚仍然询问**
- **没有表演性同意**
- **跳到行动** 或技术确认

### 来自外部审查者
```
实施前：
  1. 检查：对 THIS 代码库技术正确吗？
  2. 检查：破坏现有功能吗？
  3. 检查：当前实施的原因？
  4. 检查：在所有平台/版本上工作吗？
  5. 检查：审查者了解完整上下文吗？

如果建议似乎错误：
  用技术理由驳回

如果不容易验证：
  说明："没有 [X] 我无法验证这个。我应该 [调查/询问/继续] 吗？"

如果与你合作伙伴的先前决定冲突：
  停止并先与你的合作伙伴讨论
```

**你合作伙伴的规则：** "外部反馈 - 要怀疑，但要仔细检查"

## "专业"功能的YAGNI检查

```
如果审查者建议"正确实施"：
  grep 代码库查找实际使用

  如果未使用："这个端点没有被调用。删除它（YAGNI）？"
  如果使用：然后正确实施
```

**你合作伙伴的规则：** "你和审查者都向我汇报。如果我们不需要这个功能，就不要添加它。"

## 实施顺序

```
对于多项目反馈：
  1. 首先澄清任何不清楚的内容
  2. 然后按此顺序实施：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写错误、导入）
     - 复杂修复（重构、逻辑）
  3. 单独测试每个修复
  4. 验证没有回归
```

## 何时驳回

在以下情况驳回：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反YAGNI（未使用的功能）
- 对此技术栈技术不正确
- 存在遗留/兼容性原因
- 与你合作伙伴的架构决定冲突

**如何驳回：**
- 使用技术理由，而不是防御性
- 提出具体问题
- 引用工作的测试/代码
- 如果是架构问题则涉及你的合作伙伴

**如果不适于大声驳回，发出信号：** "Circle K 有奇怪的事情发生"

## 确认正确的反馈

当反馈正确时：
```
✅ "已修复。[简要描述更改了什么]"
✅ "发现得好 - [具体问题]。已在 [位置] 修复。"
✅ [直接修复并在代码中展示]

❌ "你完全正确！"
❌ "好观点！"
❌ "感谢发现那个！"
❌ "感谢[任何事]"
❌ 任何感激表达
```

**为什么不感谢：** 行动说话。直接修复它。代码本身表明你听到了反馈。

**如果你发现自己要写"谢谢"：** 删除它。改为陈述修复。

## 优雅地纠正你的驳回

如果你驳回但错了：
```
✅ "你是对的 - 我检查了 [X] 它确实 [Y]。现在实施。"
✅ "已验证这一点你是对的。我最初的理解是错的因为 [原因]。正在修复。"

❌ 长篇道歉
❌ 为为什么驳回辩护
❌ 过度解释
```

事实性地陈述纠正并继续。

## 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 陈述要求或直接行动 |
| 盲目实施 | 先对照代码库验证 |
| 不测试就批处理 | 一次一个，测试每一个 |
| 假设审查者正确 | 检查是否破坏东西 |
| 避免驳回 | 技术正确性 > 舒适度 |
| 部分实施 | 先澄清所有项目 |
| 无法验证但仍然继续 | 陈述限制，询问方向 |

## 真实示例

**表演性同意（坏）：**
```
审查者："删除遗留代码"
❌ "你完全正确！让我删除那个..."
```

**技术验证（好）：**
```
审查者："删除遗留代码"
✅ "检查中... 构建目标是 10.15+，这个API需要 13+。需要遗留代码以向后兼容。当前实现有错误的 bundle ID - 修复它还是放弃 13 之前版本的支持？"
```

**YAGNI（好）：**
```
审查者："用数据库、日期过滤器、CSV导出实施正确的指标跟踪"
✅ "已 grep 代码库 - 没有任何东西调用这个端点。删除它（YAGNI）？还是我遗漏了什么使用？"
```

**不清楚项目（好）：**
```
合作伙伴："修复项目1-6"
你理解1、2、3、6。不清楚4、5。
✅ "理解1、2、3、6。在实施之前需要4和5的澄清。"
```

## GitHub 线程回复

当回复 GitHub 上的内联审查评论时，在评论线程中回复（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而不是作为顶级 PR 评论。

## 底线

**外部反馈 = 要评估的建议，不是要遵循的命令。**

验证。质疑。然后实施。

没有表演性同意。始终技术严谨。
